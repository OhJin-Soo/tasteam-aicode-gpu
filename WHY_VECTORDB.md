**WHY_VECTORDB**

task : 리뷰 요약
why vector db : 리뷰 요약을 위해선 전체 리뷰를 넣어야 함. 전체 리뷰를 llm에 넣고 요약을 시키기보단, 의미 검색을 통해, 요약에 필요한 부분인 '긍정적인 측면'(맛있다, 좋았다, 친절했다) 등의 측면을 따로 선별, 또 '부정적인 측면'(맛없다, 싫었다, 불친절했다)를 따로 선별하여 각각에 대한 요약과 전체에 대한 요약이 좀 더 의도에 맞는 요약이 이루어질거라고 봤음.

task : 리뷰 강점 추출
why vector db : 리뷰 강점을 추출하기 위해선, 강점을 알아야함. 그리고, 모든 리뷰에 대한 강점을 추출해야함. 예를들어, 음식점 1개 대비 나머지 음식점 20개와 비교한다고 가정하면, 모든 음식점의 장점에 대한 요약을 전부 context로싸 이용해아함. 로직적으로 매우 복잡하다고 판단함. 따라서 target_restaurant 강점 대비 non_target_restaurant 강점으로 한번에 비교하는 방식을 택함.

task : 특정 의미 리뷰의 이미지 추출
why vector db and why don't use CLIP : 메인에 이미지 큐레이션 추천이 필요했음. 의미를 반영한 이미지 추천을 위해 Vectordb를 사용. 이미지가 있는 리뷰에 한정해서, 원하는 의미의 텍스트를 검색하고, 그 리뷰를 반환하는 방법을 택함. CLIP을 통한 이미지와 리뷰 텍스트간 유사도를 파악하여 이 이미지가 리뷰와 상관 있는지 검증하는 방법도 존재는 하지만, 꼭 이미지와 리뷰 텍스트가 유사하다고 음식점 추천 용도로 적합한 건 아니다. 즉, CLIP은 보통 1:1 매칭을 원하는데, 분위기가 좋다는 리뷰지만, 음식 사진을 올 수도 있는 법이다. 예를 들어, 사람들은 분위기가 좋다라고 텍스트엔 적었지만, 사진은 친구와 시킨 음식 사진을 찍을수도 있는법이라고 생각해. 메인 큐레이션의 목적은 어차피 추천이기 때문에, 비록, 제목은 분위기 좋은 음식점 이지만, 사진은 음식 사진일수도 있다고 생각해. 그리고 이건 흔하다고 생각함. 근데, 이게 추천 목적인 메인 큐레이션에 맞다고 생각. clip은 1:1 매칭이므로 텍스트와 이미지가 유사한 경우를 많이 못찾을 가능성이 높다고 생각. 따라서 차라리 human-in-the-loop를 도입해, 최종적으로 메인 큐레이션에 올리기 전에 이게 메인 큐레이션에 해당 메인 큐레이션의 컨셉('분위기 좋은 음식점')에 벗어나는지 여부를 사람이 직접 판단하여 메인 큐레이션에 노출하자는 생각.
